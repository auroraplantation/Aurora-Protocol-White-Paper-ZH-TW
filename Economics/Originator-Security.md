# 發起人擔保

Aurora Protocol 透過 $AUR 治理代幣實作**發起人質押要求**，以對齊發起人激勵與平台績效，並為 DeFi 參與者提供首損緩衝。

---

## 激勵問題

在農業融資中，發起人預先獲得資金並於日後還款。這產生了時間不對稱性：參與者自投入時刻起即承擔資金風險，而發起人的義務則被遞延。缺乏制衡機制的情況下，此不對稱性有利於可能表現不佳、延遲還款或違約的發起人。

傳統金融透過擔保品要求、信用評分和法律執行來處理此問題。Aurora Protocol 在其[驗證架構](../Protocol/Verification.md)之外，以鏈上經濟機制作為補充：**$AUR 質押**。

---

## 運作方式

### 質押門檻

發起人在 Aurora Protocol 上部署批次前，須質押一定數量的 $AUR 代幣。質押金額在批次生命週期內鎖定，功能如下：

1. **切身利害關係** — 發起人對批次成敗具有直接的經濟曝險
2. **首損緩衝** — 違約事件中，質押的 $AUR 可被清算以部分補償參與者
3. **品質信號** — 願意質押代幣彰顯發起人對自身營運的信心

### 質押參數

| 參數 | 說明 |
|---|---|
| 質押代幣 | $AUR（Aurora 治理代幣） |
| 質押金額 | 依批次定義，與融資金額成比例 |
| 鎖定期 | 批次生命週期全程（從部署至里程碑 3 完成或失敗處理） |
| 解鎖條件 | 回報全額存入 ClaimVault |
| 罰沒條件 | 批次進入失敗狀態且存在未釋放的託管資金 |

### 生命週期

```
   發起人質押 $AUR
           │
           ▼
   批次部署 ──── 批次成功 ──── $AUR 返還發起人
           │
           └──── 批次失敗 ──── $AUR 罰沒（部分或全部）
                                      │
                                      ▼
                              罰沒所得按比例分配予
                              受影響參與者
```

---

## 罰沒機制

若批次轉入**失敗**狀態：

1. 發起人質押的 $AUR 被標記為罰沒
2. 罰沒的 $AUR 透過既定流程清算（兌換為 USDC）
3. 清算所得按比例分配予受影響的代幣持有者
4. 罰沒依智能合約邏輯自動執行，無需人工介入

### 罰沒限制

質押的 $AUR 提供**部分**損失緩解，而非完整保險。可追回金額取決於：

- 質押比例相對於批次規模
- 清算時 $AUR 的市場價格
- 失敗前已釋放予發起人的託管資金比例

Aurora Protocol 不保證罰沒所得能覆蓋參與者損失。詳見[風險揭露](../Risks.md)。

---

## 質押作為信譽

除經濟功能外，發起人質押建立了**信譽信號**：

- 擁有成功批次紀錄的發起人累積解鎖的 $AUR，彰顯可靠性
- 重複合作的發起人可獲得更優條件（更低質押比例、更大批次額度），因其鏈上歷史紀錄持續積累
- 首次合作的發起人須滿足較高質押門檻，以補償缺乏歷史績效資料的風險

這創造了從高門檻、高擔保的早期批次到更高效、基於關係的融資之間的自然過渡。

---

## $AUR 代幣狀態

$AUR 治理代幣**尚未發行**。計劃作為獨立的未來代幣發行，與 RWA 批次代幣不同。主要資訊如下：

- $AUR 將作為 Aurora Protocol 生態系的治理及功能代幣
- $AUR 作為[Aurora 積分計畫](Points-Program.md)的**空投權重**——參與者累積的積分將在未來代幣生成事件中轉換為 $AUR 分配
- 發起人質押機制將於 $AUR 上線時啟用

在 $AUR 發行前，發起人擔保透過鏈下協議、驗證流程及 Aurora Labs 的營運監督進行管理。

## 設計考量

| 替代方案 | 排除原因 |
|---|---|
| USDC 擔保品 | 減少發起人的可用營運資金；違背提供融資的初衷 |
| 僅基於 NFT 的信譽 | 對不良表現無經濟後果 |
| 第三方保險 | 費用高昂、流程緩慢，且多數東南亞農業操作無法取得 |
| **$AUR 質押** | **對齊激勵、產生經濟後果、建立信譽，並連結至協議治理** |
