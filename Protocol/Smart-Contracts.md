# 智能合約

Aurora Protocol 的鏈上基礎設施由**五個核心智能合約**組成，共同治理批次融資操作的完整生命週期。所有合約部署於 **Ethereum 主網**，以 **Solidity ^0.8.20** 編寫，基於 **OpenZeppelin** 標準函式庫構建。

---

## 架構概覽

```
                    ┌──────────────┐
                    │ BatchFactory │
                    └──────┬───────┘
                           │ 部署（最小代理克隆）
              ┌────────────┼────────────┐
              ▼            ▼            ▼
      ┌──────────────┐ ┌───────────┐ ┌────────────┐
      │ AuroraRWA1155│ │PrimarySale│ │ EscrowVault│
      └──────────────┘ └───────────┘ └─────┬──────┘
                                           │
                                    ┌──────▼──────┐
                                    │  ClaimVault  │
                                    └─────────────┘
```

每次批次部署均建立一組新的關聯合約實例。採用**最小代理克隆模式**（EIP-1167）以最小化 Gas 成本——每個新實例委託至預先部署的實作合約，而非重新部署完整位元組碼。

---

## 合約規格

### 1. BatchFactory

工廠合約是批次建立的唯一入口。它協調每個新批次中四個子合約的原子化部署。

**職責：**
- 部署 AuroraRWA1155、PrimarySale、EscrowVault 和 ClaimVault 的最小代理克隆
- 在初始化期間將四個合約相互關聯
- 儲存所有已部署批次的登錄表
- 強制執行部署時參數（融資目標、代幣供應量、里程碑計劃、費率）

**權限：** 僅限授權部署者（Aurora Labs 管理員角色）。

### 2. AuroraRWA1155

代幣合約實作 **ERC-1155** 多代幣標準，代表批次中的分割式參與權。

**職責：**
- 在初級銷售期間鑄造 RWA 代幣
- 追蹤分割式批次參與權的所有權和餘額
- 支援領取流程中的銷毀操作
- 儲存批次層級的中繼資料（URI、批次參數）

**設計考量：** 選擇 ERC-1155 而非 ERC-20 或 ERC-721 的原因如下：
- 單一合約可代表多種代幣類型（為未來多層級批次預留彈性）
- 原生批次轉帳支援降低了組合操作的 Gas 成本
- 透過代幣 URI 標準實現中繼資料的可擴展性

**現行狀態：** 目前不支援二級市場交易。代幣在當前版本中為不可轉讓（合約層級強制限制轉讓）。二級市場功能規劃於未來協議升級中實現。

### 3. PrimarySale

銷售合約管理 RWA 代幣向參與者的初始分配。

**職責：**
- 接收參與者的 USDC 存入
- 透過 AuroraRWA1155 鑄造對應的 RWA 代幣
- 強制執行融資窗口（開始時間、結束時間）
- 強制執行融資目標（最低及最高募集金額）
- 銷售成功完成後將募集資金轉入 EscrowVault
- 若未達融資目標，將資金退還參與者

**計價單位：** 所有初級銷售以 USDC（ERC-20 穩定幣）計價。

### 4. EscrowVault

託管合約是核心資金管理機制。它持有所有批次資金，並依據預設的里程碑計劃向發起人釋放資金。

**職責：**
- 接收來自 PrimarySale 合約的資金
- 管理治理批次生命週期的**七狀態有限狀態機**
- 於授權確認後向發起人釋放里程碑對應的資金批次
- 處理失敗轉換和參與者退款
- 在釋放時點扣除平台費用

詳見 [EscrowVault](EscrowVault.md) 完整狀態機規格。

### 5. ClaimVault

領取合約管理批次完成後向代幣持有者的收益分配。

**職責：**
- 接收發起人或 EscrowVault 的回報存入（本金 + 收益）
- 處理銷毀領取贖回：代幣持有者銷毀 RWA 代幣並收取按比例的 USDC
- 確保總存入與總領取之間的精確會計
- 透過銷毀機制防止重複領取（已銷毀代幣無法重複使用）

詳見[銷毀領取](Burn-to-Claim.md)完整贖回機制。

## 安全模型

### 角色基礎存取控制

所有合約均實作 OpenZeppelin 的 **AccessControl** 模組，包含以下角色：

| 角色 | 範圍 | 指派對象 |
|---|---|---|
| `DEFAULT_ADMIN_ROLE` | 完整管理權限 | Aurora Labs 多簽錢包（移交後） |
| `DEPLOYER_ROLE` | 透過 BatchFactory 建立批次 | Aurora Labs 部署者 |
| `OPERATOR_ROLE` | 里程碑確認、狀態轉換 | Aurora Labs 操作者 |
| `PAUSER_ROLE` | 緊急暫停功能 | Aurora Labs 管理員 |

### 部署後管理權移交

初始部署與設定完成後，部署者帳戶放棄其提升的權限，並將管理控制權轉移至指定的多簽錢包。這確保系統上線後，任何單一金鑰均無法單方面修改合約狀態或存取控制。

### 升級路徑

當前合約為**不可升級**。每個批次部署一旦建立即為不可變。協議層級的變更透過新的實作合約引入——未來批次將從新實作克隆，而既有批次繼續在其原始邏輯下運行。

## 審計狀態

智能合約**尚未經過**第三方安全公司審計。全面審計計劃於生產批次主網部署前完成。詳見[風險揭露](../Risks.md)中有關未審計合約風險的說明。
