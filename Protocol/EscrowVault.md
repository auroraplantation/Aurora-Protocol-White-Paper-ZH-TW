# EscrowVault

EscrowVault 是 Aurora Protocol 的核心資金管理合約。它實作了一個**七狀態有限狀態機**，治理批次資金的完整生命週期——從初始存入、里程碑逐階段撥付，到最終結算或失敗處理。

---

## 狀態機

```
                 ┌────────┐
                 │  空白   │
                 └───┬────┘
                     │ PrimarySale 存入資金
                     ▼
                 ┌────────┐
           ┌─────│ 融資中  │─────┐
           │     └───┬────┘     │
           │         │ 達標     │ 未達標 /
           │         │          │ 窗口到期
           │         ▼         │
           │     ┌────────┐     │
           │     │ 已融資  │     │
           │     └───┬────┘     │
           │         │ M1       │
           │         │ 確認     │
           │         ▼         │
           │   ┌───────────┐    │
           │   │ 里程碑 1   │    │
           │   └─────┬─────┘    │
           │         │ M2       │
           │         │ 確認     │
           │         ▼         │
           │   ┌───────────┐    │
           │   │ 里程碑 2   │    │
           │   └─────┬─────┘    │
           │         │ M3       │
           │         │ 確認     │
           │         ▼         │
           │   ┌───────────┐    │
           │   │ 里程碑 3   │    │
           │   └───────────┘    │
           │                    │
           │     ┌────────┐     │
           └────►│  失敗   │◄────┘
                 └────────┘
```

## 狀態定義

| 狀態 | 說明 |
|---|---|
| **空白（Empty）** | 初始狀態。合約已部署但未包含任何資金。 |
| **融資中（Funding）** | PrimarySale 進行中，參與者正在存入資金。 |
| **已融資（Funded）** | 融資目標已達成，銷售關閉。資金鎖定，等待首個里程碑確認。 |
| **里程碑 1（Milestone 1）** | 第一批資金已釋放給發起人（如播種階段資金）。 |
| **里程碑 2（Milestone 2）** | 第二批資金已釋放（如生長維護階段資金）。 |
| **里程碑 3（Milestone 3）** | 最後一批資金已釋放。批次生命週期完成，等待發起人回款。 |
| **失敗（Failed）** | 終態。因融資不足、發起人違約或營運失敗觸發。 |

## 狀態轉換

### 有效轉換

| 來源 | 目標 | 觸發條件 |
|---|---|---|
| 空白 | 融資中 | PrimarySale 合約啟動批次 |
| 融資中 | 已融資 | 於銷售窗口內達到融資目標 |
| 融資中 | 失敗 | 銷售窗口到期且未達目標 |
| 已融資 | 里程碑 1 | 操作者確認 M1 證據 |
| 已融資 | 失敗 | 發起人違約或批次取消 |
| 里程碑 1 | 里程碑 2 | 操作者確認 M2 證據 |
| 里程碑 1 | 失敗 | 發起人於 M1 階段違約 |
| 里程碑 2 | 里程碑 3 | 操作者確認 M3 證據 |
| 里程碑 2 | 失敗 | 發起人於 M2 階段違約 |

### 無效轉換

以下轉換由合約明確禁止：

- **空白 → 失敗** — 無資金的批次不可能失敗；僅維持未初始化狀態。
- **里程碑 3 → 失敗** — 所有里程碑完成後，批次即為成功。失敗不再是有效狀態。
- **失敗 → 任何狀態** — 失敗為終態，不可恢復或重啟。
- **任何反向轉換** — 狀態嚴格單向遞進，批次不可從里程碑 2 退回里程碑 1。

## 資金釋放機制

每次里程碑轉換均觸發按比例的資金釋放。釋放計劃於批次建立時定義，並編碼於合約中。

**三里程碑釋放計劃範例：**

| 里程碑 | 批次比例 | 累計釋放 |
|---|---|---|
| 里程碑 1 | 30% | 30% |
| 里程碑 2 | 35% | 65% |
| 里程碑 3 | 35% | 100% |

每次釋放時：
1. 從該批次金額中扣除平台費用（上限 3%）
2. 淨額轉入發起人指定錢包
3. 狀態轉換連同時間戳記錄於鏈上

---

## 失敗處理

當批次轉入**失敗**狀態時：

1. EscrowVault 中的**剩餘資金**開放參與者提領
2. **各參與者**可呼叫提領函式，按比例取回其在剩餘餘額中的份額
3. **先前里程碑已釋放的資金**無法透過智能合約追回——須透過鏈下追償程序及發起人質押罰沒處理

參與者在失敗情境下的可追回金額為：

```
可追回金額 = (參與者代幣數 / 總代幣供應量) × 剩餘合約餘額
```

## 存取控制

| 操作 | 所需角色 |
|---|---|
| 初始化合約 | BatchFactory（僅限部署時） |
| 轉換：融資中 → 已融資 | 自動（由 PrimarySale 觸發） |
| 轉換：已融資 → 里程碑 N | `OPERATOR_ROLE` |
| 轉換：任何 → 失敗 | `OPERATOR_ROLE` |
| 提領（失敗時） | 任何代幣持有者 |
| 緊急暫停 | `PAUSER_ROLE` |

---

## 安全特性

- **無手動資金提取** — 不存在繞過狀態機邏輯的管理員提取功能。
- **重入保護** — 所有資金轉移函式均受 OpenZeppelin `ReentrancyGuard` 保護。
- **單向狀態流** — 狀態機強制嚴格正向轉換，防止重放或回退攻擊。
- **不可變計劃** — 里程碑金額和釋放比例於部署時設定，初始化後不可修改。
