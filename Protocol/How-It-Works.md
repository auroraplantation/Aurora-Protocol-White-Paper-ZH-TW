# 運作方式

Aurora Protocol 透過**批次融資生命週期**連結 DeFi 資金提供者與東南亞農業生產者。每個批次代表一筆獨立的融資操作，具有明確的時間軸、資產類型和收益結構。

---

## 批次生命週期

### 第一階段 — 發起

農業發起人——通常為合作社、聚合商或農場經營者——向 Aurora Labs 提交融資申請。申請內容包括：

- 作物類型、地理位置及生產時間表
- 所需融資金額（以 USDC 計價）
- 預期收益及還款計劃
- 里程碑結構（通常對應播種、生長及收穫週期）

Aurora Labs 透過[五層驗證架構](Verification.md)執行盡職調查，評估發起人的歷史紀錄、營運能力及底層農業操作的可行性。

### 第二階段 — 批次部署

經核准後，Aurora Labs 透過 **BatchFactory** 合約在鏈上部署新批次。此單一交易原子化地建立以下元件：

- 新的 **AuroraRWA1155** 代幣合約，代表該批次的分割式參與權
- 關聯的 **PrimarySale** 合約，負責代幣分配
- 專屬 **EscrowVault**，按里程碑計劃持有及釋放資金
- 關聯的 **ClaimVault**，負責完成後的收益分配

每個批次均為完全獨立的鏈上實體，批次之間不存在交叉擔保或共享狀態。

### 第三階段 — 初級銷售

批次進入融資窗口期，DeFi 參與者可透過 PrimarySale 合約購買 ERC-1155 RWA 代幣。核心機制如下：

- **計價單位：** 所有交易以 USDC 計價
- **分割式參與：** 每枚代幣代表該批次的等額分割份額
- **固定價格：** 代幣價格於批次建立時設定，銷售期間不變
- **融資目標：** 批次設有明確的目標金額；達標或窗口期到期時銷售關閉

若融資窗口期內未達目標，批次轉入**失敗**狀態，所有資金透過 EscrowVault 退還參與者。

### 第四階段 — 里程碑執行

完成融資後，EscrowVault 依序經歷里程碑狀態轉換：

```
已融資 → 里程碑 1 → 里程碑 2 → 里程碑 3
```

每個里程碑階段：

1. 發起人提交里程碑完成證據（如播種確認、生長檢查、收穫文件）
2. Aurora Labs 透過驗證流程審核證據
3. 確認後，智能合約釋放對應批次的資金給發起人

此階段性釋放機制確保資金僅在實際進度獲得驗證後才撥付。詳見 [EscrowVault](EscrowVault.md) 完整狀態機規格。

### 第五階段 — 收益分配（銷毀領取）

所有里程碑成功完成後：

1. 發起人將約定的回報金額（本金 + 收益）存入 **ClaimVault**
2. EscrowVault 達到終態（里程碑 3 完成）
3. RWA 代幣持有者透過 ClaimVault 銷毀代幣，領取按比例分配的 USDC

銷毀領取機制確保清晰、確定性的結算：一枚代幣對應一份權利，銷毀即清償。詳見[銷毀領取](Burn-to-Claim.md)。

## 失敗處理

若批次在融資後的任何階段失敗——因作物歉收、發起人違約或不可抗力——EscrowVault 轉入**失敗**狀態。此情境下：

- EscrowVault 中未釋放的剩餘資金開放參與者提領
- 先前里程碑已撥付給發起人的資金須透過鏈下追償程序處理
- 發起人質押的 $AUR 擔保品（詳見[發起人擔保](../Economics/Originator-Security.md)）可能被罰沒，作為部分損失補償機制

Aurora Protocol 不保證本金返還。每個批次承載獨立風險，參與者應就每個批次獨立評估。

## 參與角色

| 角色 | 說明 |
|---|---|
| **DeFi 參與者** | 購買 RWA 代幣資助批次；批次成功完成後獲得收益 |
| **發起人** | 尋求融資的農業生產者或聚合商；須通過驗證並質押 $AUR |
| **Aurora Labs** | 平台營運方；部署批次、管理驗證、營運前端介面 |
| **智能合約** | 自主執行鏈上資金流向、狀態轉換和領取的合約系統 |

---

## 核心特性

- **非託管：** Aurora Labs 從不持有參與者資金。所有資金存放於智能合約中。
- **確定性：** 資金釋放遵循預設規則，無手動覆寫合約狀態的能力。
- **隔離性：** 每個批次經濟獨立，單一批次的失敗不會波及其他批次。
- **透明性：** 所有狀態轉換、資金流動和領取操作均可在 Ethereum 上檢視。
